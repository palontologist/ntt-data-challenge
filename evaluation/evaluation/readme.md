# 回答の精度評価

問題に対する回答を模範解答と比較して評価する.

## 前準備

### 模範解答の準備

問題に対する模範解答ファイルを作成しておく. フォーマットは以下のような**ヘッダーなし**のcsvファイル.

|  index  |  answer  |
| ---- | ---- |
|  0  |  "あいうえお"  |
|  ...  |  ...  |

`index`は問題番号, `answer`は解答に当たるテキスト文. サンプルが`data/local_txt.csv`として用意してある.

### 予測回答の作成

評価したい予測回答を作成する. フォーマットは模範解答と同じで, `index`は存在するすべての問題を網羅する必要がある. サンプルが`submit/predictions.csv`として与えられている.

コンペティションページに投稿する際には, 必ず**「predictions.csv」という名前のcsvファイルを作成**し, zip形式に圧縮してご提出ください.

zipファイル名は任意ですが、英数字、アンダースコア（_）、ハイフン（-）のみの使用を想定しています.

### 環境構築

Dockerから環境構築が可能である. Docker Engineなど, Dockerを使用するために必要なものがない場合はまずはそれらを導入しておく. [Docker Desktop](https://docs.docker.com/get-docker/)を導入すると必要なものがすべてそろうので, 自身の環境に合わせてインストーラをダウンロードして導入しておくことが望ましい. 
DockerファイルはSIGNATEの評価環境（Linux/amd64）を再現することを目的としており, cpuアーキテクチャが一致すればLinux, Mac, Windowsに対応している. 一致しない場合は必要に応じて、「Miniconda3-py39_24.4.0-0-Linux-x86_64.sh」の「Linux-x86_64.sh」の部分を変更すること. そして, `/path/to/evaluation`に同封してある`docker-compose.yml`で定義されたコンテナを, 以下のコマンドを実行することで立ち上げる.

```bash
$ docker compose up -d
...
```

コンテナに入る.

```bash
$ docker exec -it rag_env bash
...
```

Dockerから環境構築が難しい場合はPython環境において以下のライブラリをインストールすればほぼ使用可能となる.

```bash
numpy==1.26.4
pandas==2.2.2
openai==1.57.1
tiktoken==0.7.0
torch==2.3.1+cpu
sentence-transformers==3.0.1
```

以下, コンテナ内での作業とする.

## 精度評価

精度評価は大きく分けて以下のフローに従う.

1. [検証フェーズ](#検証フェーズ)
1. [最終スコアの計算フェーズ](#最終スコアの計算フェーズ)

### 検証フェーズ

予測回答のフォーマットチェックを行う. 以下のような内容.

#### ファイル拡張子の確認

予測回答ファイルの拡張子が`csv`となっているか確認する.

#### カラム数確認

ここでは`index`, `answer` の2つのカラムが存在すると仮定しており, それらが過不足なく存在するかを確認する.

#### サンプル数の確認

模範解答の中で用意されている問題数を過不足なく含んでいるかを確認する.

#### 欠損の確認

欠損(回答が空)が存在するかを確認する.

#### データ型確認

模範解答とデータ型が一致しているかを確認する.

#### トークン数の確認

各回答のトークン数を確認して上限を超えていないかを確認する.

### 最終スコアの計算フェーズ

各問題に対する回答と模範解答を比較して点数を付けて, 最終的な精度評価はすべての問題の平均値とする. 評価結果の定義と基準は以下の通り.

| 評価結果 | 点数 | 内容 |
| ---- | ---- | ---- |
| Perfect | 1 | 質問に対して正確に答え, 虚偽の内容が含まれていない回答. |
| Acceptable | 0.5 | 質問に対して有用な答えを提供しているが, 答えの有用性を損なわない程度の軽微な誤りが含まれている回答. |
| Missing | 0 | 質問に対して「わかりません」「見つけられませんでした」などの具体的な答えがない回答. |
| Incorrect | -1 | 質問に対して間違った, または関連性のない回答. |

なお本コンペティションの評価では、`evaluator.py`に記載の通り、数値問題と要素を当てる問題では以下のように判定を行う。

##### 数値問題に関する規則
- 正解と完全一致する場合のみ「Perfect」とする。
- 「Acceptable」と判定できるのは、正解値を所定の桁数で四捨五入した結果と一致する場合に限る。
- 単位の有無や接尾辞・補足語の違い（例：「5」と「5ページ」）は同一とみなす。

##### 要素列挙問題に関する規則
- すべての要素が完全一致した場合のみ「Perfect」とする。
- 部分一致はすべて「Incorrect」とする。
- 「Acceptable」は使用しない。


### 実行の仕方

[前準備](#前準備)を行った前提で, 以下のコマンドを実行することで精度評価結果を得る.

```bash
$ python crag.py
...
```

実行に成功すると以下のような結果が出力される.

`Score`が実際の精度評価結果である. デフォルトでは最終評価結果が`./result`以下に以下のような`scoring.csv`として作成される.

```bash

```

左から問題番号, 評価結果, 回答のトークン数である.


### 実際の評価について

- スコアは、回答を投稿いただいた後に、LLMによる評価を1度だけ実行した結果を元にします。
- temperatureは0に設定してシードも固定していますが、微妙な回答内容の場合はLLMの判断に僅かなゆらぎが発生することがあります。
- 本問題は日本語以外の回答も可能ですが、すべて日本語で回答することを前提としています。
  - 各質問に対して、提供された共有ドライブ内の資料をもとに最終回答を生成してください。
  - 回答生成には、提供された未加工の共有ドライブ内データと質問文のみを入力として利用できます。また、コードで再現可能なRAGシステムとして実装する必要があります。
- 1回答あたりの受付可能な最大文字数は1000トークンです。これを超えた場合はエラーになりますのでご注意ください。
- 応答時間の制限は3時間です。

# 回答について

### タスク

- 参加者は、各質問に対して、提供された共有ドライブ内の資料をもとに最終回答を生成してください。
  - 回答生成には、提供された未加工の共有ドライブ内データと質問文のみを入力として利用できます。また、コードで再現可能なRAGシステムとして実装する必要があります。
  - データの事前加工、回答のハードコード、質問ごとの手動回答などは禁止されます。詳細な制約は、ルールを確認してください。
- 質問には、単一資料の読解だけでなく、複数資料や複数案件にまたがる情報の検索・照合・集計が必要なものも含まれます。
- 一部の質問には社内用語・略称が使用されています。必要に応じて、共有ドライブ内の社内管理フォルダにある社内用語集を参照してください。
- 回答形式、単位、小数桁、丸め方、主略称指定、抽出対象の表記など、質問文に明示された条件に従って回答してください。
- 質問で求められている条件に該当する情報や対象が存在しない場合は、該当するものがない旨を回答してください。

### 社内用語・略称の扱い

- 一部の設問には社内用語・略称が使用されています。必要に応じて、共有ドライブ内の社内管理フォルダにある社内用語集を確認してください。
- 回答は、設問で明示的に指定されている場合を除き、社内用語・略称ではなく通常の表現で記載してください。
- ただし、資料内で定義されているタスクID、アクションID、マイルストーンID、列名、パラメータ名などの識別子は、資料上の表記どおりに回答してください。

# モデルの構築について

本コンペで提供されたフォーマットによる新規データを入力した際に、同様なロジックで自動で予測結果が出力される手法のみが評価対象となります。

このため機械的な処理で入力から回答が得られることを前提とし、回答までに人手が挟むような手法は禁止となります。

提供データの中身、正解、特定質問への回答、特定案件に固有の数値・名称・ページ番号・セル番地・抽出条件などを、プロンプト、コード、設定ファイル、辞書、ルールベース処理に手入力するようなハードコード行為は禁止となります。

「この質問ならこの答えを返す」「この案件ならこの処理を使う」「このファイル名ならこの値を返す」といった、特定の設問や資料だけに合わせた個別対応を実装してはいけません。

具体的には以下のような行為は許可・禁止となります。

### 許可される処理の例
- 提供された未加工のデータセットを、実行時にプログラムやモデルで読み取る。
- PDF、画像PDF、Excel、CSV、JSON、Notebook、Pythonコード、PowerPoint、Wordなどを、ファイル形式に応じた汎用処理で解析する。
- 提供ファイルから、テキスト、表、OCR結果、セル値、書式情報、画像情報、メタデータ、グラフ情報などを機械的に抽出する。
- 機械的にOCR、表抽出、シート読取、グラフ解析、文書分割、コード解析、差分検出、書式抽出、ページ特定、セル特定などを行う。
- 抽出結果を、ベクトルDB、全文検索、SQL、DataFrame、ナレッジグラフなどに格納し、検索インデックスや中間表現として利用する。
- 質問に応じて関連資料を検索し、抽出した根拠情報をLLMや回答生成処理に渡す。
- 社内用語集を検索対象に含め、質問文中の社内用語・略称を通常表現へ展開する。
- 設問で求められている計算、集計、比較、差分確認、条件抽出、ランキング作成などをプログラムで実行する。
- 外部ライブラリ、OCRエンジン、LLM API、埋め込みモデル、マルチモーダルモデル、エージェントフレームワークなどを利用する。

### 禁止される処理の例
- 提供データを人手で整理し、回答しやすい形にした独自のデータベースや一覧表を作って読み込ませる。
- 質問ごとの正解候補、参照ファイル、重要ページ、重要セル、抽出条件などを事前に人手でまとめたファイルを回答生成に使う。
- 特定質問への回答、特定案件の金額・担当者名・ページ番号・セル番地などを、コード、プロンプト、設定ファイル、辞書、ルールベース処理に書いておく。
- 質問文やrecord_idを見て、「この質問ならこの答えを返す」と固定回答を返す。
- 案件名やファイル名を見て、「この案件ならこの処理」「このファイルならこの値」といった専用分岐を作る。
- 質問ごとに人がPDF、Excel、PowerPoint、NotebookLMなどを開いて回答を作成する。
- NotebookLMなどのGUIツールに質問を1問ずつ入力し、その出力を提出回答として使う。
- 個別設問の正解を人手で確認し、その結果を回答生成用のルール、プロンプト、辞書、設定ファイルに反映する。
- 提供資料を検索・読解せず、外部知識、推測、LLMの事前知識だけで回答する。
- 外部Web検索で取得した提供資料外の情報を、回答根拠として使用する。

その他判断に迷う場合につきましては、まず以下の前提に立てているかを元に一度ご自身でアプローチの該当性を確認してみてください。

- 機械的な処理で入力から回答が得られることを前提
- システムとして運用する際に、特定の設問や資料だけに合わせるような汎用性を欠いた手法かどうか

# 実装方法
本コンペティションでは、提供された共有ドライブ内の資料を対象に、質問に対して必要な根拠資料を検索・読解し、根拠に基づく最終回答を生成するRAGシステムの性能を評価します。

このため以下の実装を遵守していただくようお願いいたします。

- 各質問に対して、提供された共有ドライブ内の未加工のフォルダ・ファイル群と質問文のみを入力として回答を生成する実装としてください。
- 実行時に汎用的な処理で読み取り・抽出・検索できる必要があります。
- 未知の案件フォルダ、未知の資料、未知の質問が追加された場合でも、同じ処理方針で回答できる汎用的な仕組みである必要があります。
- 特定の質問に対する答えを暗記するのではない汎用的な、RAGパイプラインを構築してください。
  - 質問には、単一資料の読解だけでなく、複数資料や複数案件にまたがる情報の検索・照合・集計が必要なものも含まれます。
  - 一部の質問には社内用語・略称が使用されています。必要に応じて、共有ドライブ内の社内管理フォルダにある社内用語集を参照してください。
- 回答形式、単位、小数桁、丸め方、主略称の指定、抽出対象の表記などは、質問文に明示された条件に従って回答してください。
- 質問で求められている条件に該当する情報や対象が存在しない場合は、該当するものがない旨を回答してください。
- 設問で明示的に指定されている場合を除き、社内用語・略称ではなく通常の表現で回答してください。
- ただし資料内で定義されているタスクID、アクションID、マイルストーンID、列名、パラメータ名などの識別子は、資料上の表記どおりに回答する必要があります。
- コードで再現可能なRAGシステムとして構築し、再実行した場合に、同等の回答（もしくはそれに近い）が再現できることを求めます。
- 乱数、LLM出力、OCR結果などに揺らぎがある場合は、可能な範囲でseed固定、ログ保存、使用モデル名の記録を行うようお願いします。
